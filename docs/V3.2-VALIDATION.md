# V3.2 Validation

## Goal

V3.2 adds a candidate bucket system on top of the stable V3.1 ranking chain.
It links:

- `final_action`
- `decision_rank_score`
- `ml_score`
- `final_rank_score`

and produces:

- `candidate_bucket`
- `bucket_reason`
- `bucket_priority`

This layer does not modify `final_action`.

## Bucket Rules

First version buckets:

- `EXECUTE`
- `CANDIDATE`
- `REVIEW`

Default policy:

- `EXECUTE`
  - `final_action in ('ENTER', 'ADD', 'HOLD')`
  - `final_rank_score >= 0.65`
  - no blocking hard constraints
  - data quality not blocked
- `CANDIDATE`
  - executable or reviewable action with mid score
  - score in the middle band
  - may include data quality warning but not blocker
- `REVIEW`
  - `final_action` is already conservative (`REVIEW`, `AVOID`, `EXIT`)
  - or data quality is below minimum
  - or hard constraints block execution
  - or final score is too low

Priority:

- `EXECUTE = 1`
- `CANDIDATE = 2`
- `REVIEW = 3`

Sorting order:

1. `bucket_priority ASC`
2. `final_rank_score DESC`
3. `decision_rank_score DESC`

## Validation Matrix

| Scenario | Expected |
| --- | --- |
| High score `ENTER/HOLD` | `EXECUTE` |
| Mid score `ENTER/HOLD` | `CANDIDATE` |
| `REVIEW` action | `REVIEW` |
| Poor data quality | `REVIEW` |
| No ML score | bucket still assigned, `final_rank_score` falls back to `decision_rank_score` |
| Mixed batch | `EXECUTE` before `CANDIDATE`, `CANDIDATE` before `REVIEW` |

## Key Samples

- `ENTER + final_rank_score=0.82 + no blockers -> EXECUTE`
- `HOLD + final_rank_score=0.58 -> CANDIDATE`
- `ENTER + insufficient bars -> REVIEW`
- `REVIEW + any score -> REVIEW`

## Known Boundaries

- No database schema change in V3.2
- No new tables
- No automatic trade execution
- `decision_policy.py` remains the only action decision exit
- `ranking.py` may classify and sort, but must not change `final_action`
- `runner.py` only passes through bucket fields
